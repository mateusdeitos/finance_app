# Milestone v1.5 — Import Transactions Performance — Retrospective

**Started:** 2026-05-05
**Completed:** 2026-05-07 (3 days)
**Phases shipped:** 16, 17, 18, 19, 21 (P20 skipped by gate decision)
**Branch:** `claude/optimize-import-performance-ouNOx`

## TL;DR

Started with a 100-row CSV import flow where every keystroke triggered a 700–900 ms commit and every checkbox click cascaded through all 100 rows. After 4 measure-then-fix phases the same flow now reacts in <6 ms for keystrokes and re-renders only the rows that legitimately changed for selection actions. Profile-driven, not speculative.

## Headline trajectory

| # | Scenario              | P16 baseline | P19 final |    Speedup |  Gate (16ms) |
|---|-----------------------|-------------:|----------:|-----------:|:-------------|
| 1 | Description keystroke |     761 ms   |  3.5 ms   |  **218×**  | ✅ way below |
| 2 | Amount keystroke      |     929 ms   |  5.6 ms   |  **166×**  | ✅ way below |
| 3 | Checkbox toggle       |     791 ms   | 169.9 ms  |    4.7×    | ⚠️ above, but only the toggled row re-renders |
| 4 | Shift-click 0..50     |     720 ms   | 466.4 ms  |    1.5×    | ⚠️ above, but 49 rows legitimately changed state |

All numbers are raw dev-mode (StrictMode 2× + HMR overhead). Production estimate ≈ ÷2.

**Composition is what matters most:** every action now re-renders exactly the components that changed. No cascade. The gate is met for the high-frequency interactions (keystrokes); the remaining cost on selection actions is intrinsic Mantine internals, not waste.

## Phase-by-phase deltas

### Phase 16 — Baseline + diagnostics (2026-05-05 → 06)
- Built deterministic 100-row CSV fixture (`scripts/genImportFixture.ts`) + centralized `buildCsvContent` helper
- Empirically confirmed `babel-plugin-react-compiler` was INACTIVE in the production bundle (locked finding)
- Captured 4 reproducible profiler scenarios; verified hypothesis: **page-level `useWatch({ name: 'rows' })` was the dominant trigger for keystrokes**
- Locked the `just profile` recipe (Vite dev server) after the production build's profiling-instrumentation strip blocked profiler access
- Surfaced 2 secondary findings (cenários 3/4 not caused by useWatch — page-level `useState<Set<number>>` for `selected` was the actual culprit; cenário 2 had an additional intra-row updater)

### Phase 17 — Eliminate page-level useWatch cascade (2026-05-06 → 07)
- Removed broad `useWatch({ name: 'rows' })` at `ImportTransactionsPage.tsx:70`
- Expanded the existing `compute`-scoped `useWatch` to return 4 scalars (`total`, `totalSuccess`, `errorCount`, `toImportPendingCount`)
- `handleSelectAll` switched from `rows.map` to `fields.map` (cardinality from `useFieldArray`)
- Compiler wiring **skipped** (user-chosen scope reduction; preserves attribution)
- Cenário 1: 761 → 2.9 ms (262×); Cenário 2: 929 → 62.5 ms (15×). Cenários 3/4 unchanged (deferred to P18)

### Phase 18 — Memoize options + rearch selection (2026-05-07)
- **Scope expanded mid-phase by user approval** to absorb the deferred selection cascade
- Added Zustand@^5 (~1.5 kB gzipped). `selectionStore.ts` keyed by `field.id` (stable from `useFieldArray`); fixes a latent index-shift bug in `handleRemoveSelected`
- Per-slot subscription via `useSelectionStore(s => s.selected.has(fieldId))` so only the toggled row re-renders
- Top-level select functions in `useImportOptions.ts` (`useCategoryOptions`, `useAccountOptions`, `useSharedAccounts`) — stable identity for TanStack Query memoization
- **The ref-callback gotcha** — first measurement showed the cascade still active. Diagnosis: inline `ref={(el) => rowRefs.current.set(i, el)}` in `fields.map` was a fresh function per render, breaking `React.memo` even with `forwardRef`. Fixed via `useMemo([fields])` returning per-index callbacks. React DevTools `changeDescriptions` reason `props=['ref']` on 199 of 200 rows was the diagnostic giveaway.
- Cenário 3: 791 → 183.9 ms (4.3×); Cenário 4: 720 → 455.3 ms (1.6×). Cenário 2 unchanged (intra-row updater deferred to P19)

### Phase 19 — Scope & debounce duplicate check (2026-05-07)
- **Surprise:** the hook already had 500 ms debounce via Mantine `useDebouncedValue`. ROADMAP's "200-300ms debounce" target was outdated. The actual cenário 2 cost was structural row re-render, not network burst
- Added `enabled: action === 'import'` gating (skip/duplicate rows short-circuit)
- **Extracted `<RowDuplicateCheck>` sub-component** — null-rendering, owns the date/amount subscription internally. Row outer's `useWatch` dropped from 10 → 8 fields. **Amount keystrokes no longer re-render the row's Mantine subtree at all.**
- Removed redundant `rowAmount` prop from `<SplitPopover>` (`handleOpen` already reads via `parentForm.getValues`)
- Cenário 2: 64.4 → **5.6 ms** (11.5× faster). Below the gate.

- After P19, keystroke axes (cenários 1/2) below the 16 ms gate
- Cenários 3/4 above the gate but cost is intrinsic Mantine internals on the row(s) that legitimately need to update — not row-count waste
- 100-row system hard limit caps DOM weight regardless of virtualization
- ROI judged too low; orchestrator confirmed skip
- Remains a follow-up if user-feedback says 170ms toggle / 466ms shift-click is laggy in practice

## Decisions worth preserving

1. **Profile-then-fix > speculate-then-fix.** Every phase had measurement evidence before code changes. P19 specifically caught the wrong target ("debounce") because the existing 500ms was already in place — the real bottleneck was intra-row subscription scope.

2. **Compiler wiring deferred across the whole milestone.** Kept attribution clean: every delta is the manual refactor, not the compiler. If a future phase wires `babel-plugin-react-compiler`, its contribution will be a clean isolated delta on top of these numbers.

3. **`just profile` (Vite dev server) over `vite build --mode development`.** Initial attempt to use a "dev-mode build" was rejected — Vite's `--mode` flag controls `.env` resolution only, not `process.env.NODE_ENV` for the bundle, so React's production bundle was still emitted and the profiler stayed blocked. Documented at `16-CONTEXT.md` → addendum 2026-05-05.

4. **StrictMode 2× caveat acknowledged once, applied consistently.** All 4 phases used the same `just profile` recipe; cross-phase comparisons are internally consistent. Production estimate: divide by ~2 mentally.

5. **State keyed by stable id, not array index.** P18's selection store keys by `field.id` (from `useFieldArray`), eliminating a latent bug where `handleRemoveSelected` would have left stale entries if the post-action clear were ever removed.

6. **Sub-component extraction for narrowing watch scope.** P19's `<RowDuplicateCheck>` (null-rendering) is the canonical React pattern when a hook depends on a small subset of fields and the parent's outer `useWatch` was over-subscribing. Worth pattern-establishing for future refactors.

## Lessons learned

### The ref-callback false-negative (P18 v1)

The architecture was correct from the first commit, but the first measurement made it look broken. An inline ref callback per `fields.map` iteration was invalidating `React.memo` on all 100 rows. The diagnostic signal was specific: React DevTools `changeDescriptions` showing `props=['ref']` for 199 of 200 rows.

**Pattern to internalize:** when adding `React.memo` to a row component, audit forwarded refs and callback props for inline arrows in parent map iterations. Use `useMemo` over the array to produce one stable callback per index.

### ROADMAP targets aren't measurements

P19 SC2 said "debounce 200-300ms". Investigation found the existing hook already had 500ms debounce. The roadmap text was written at scope-time without inspecting the code. **Lesson:** revisit roadmap SCs at phase-start with a code grep — they're hypotheses to confirm, not gospel to enforce.

### "Already in place" is a valid finding

Sometimes the work for a phase is partly done. P19's debounce was already there. The actual work was elsewhere (row-watch narrowing). Recognizing that and adapting saved time vs. rewriting a working debounce.

### Profiler artifacts are large but priceless

~140MB per phase capture on disk (4 cenários × ~35MB each). Total committed: ~700MB of JSON across 5 captures (P16, P17, P18 v1 [discarded], P18 v2, P19 — though user overwrote P18 v2 files with P19, so on disk it's ~3 generations stored). The cost is real but the analytical value (composition, updaters, changeDescriptions) is what made every phase's verdict defensible.

**Pattern to internalize:** consider profiler exports the source of truth, the markdown comparison docs the human-readable summary. If a comparison doc claims a number, the JSON in `frontend/profilling/` should back it.

## Patterns established (re-usable for v1.6+)

| Pattern | Where it lives now | When to reach for it |
|---------|---------------------|----------------------|
| `compute`-scoped `useWatch` returning multiple scalars | `ImportTransactionsPage.tsx:53` | Page subscribes to derived counts but not raw arrays |
| Cardinality from `useFieldArray.fields.length` (no useWatch) | `ImportTransactionsPage.tsx:131,287` | Need length/index reads of a field array |
| Top-level select functions for `useQuery({ select })` | `useImportOptions.ts` | Per-row hooks that need stable Mantine `data` references |
| Zustand store keyed by stable id | `selectionStore.ts` | Page-level state that fan-outs to many child subscribers |
| Per-slot subscription via Zustand selector | `ImportReviewRow.tsx:64` | Children need to read their own slice without re-rendering siblings |
| `useMemo` over field array for stable per-index ref callbacks | `ImportTransactionsPage.tsx:78` | Parent renders many memoized children; refs need stable identity |
| Null-rendering sub-component for narrowing watch scope | `ImportReviewRow.tsx:493` (`RowDuplicateCheck`) | Hook depends on a small subset of fields; parent's outer `useWatch` was over-subscribing |
| `enabled` gating on background-effect hooks | `useDuplicateTransactionCheck.ts` | Hook callers can short-circuit network/effect when conditions aren't met |
| `just profile` recipe for reproducible measurements | `frontend/justfile` | Any future perf phase — same recipe = apples-to-apples |

## What worked well

- The discuss → plan → execute → verify → summary loop kept scope tight and decisions traceable
- "Skip the compiler wiring" decision early in P17 prevented confusion when later measurements changed
- User's call on Zustand for P18 was decisive — saved a sprawl of "store options" debate
- The ref-callback diagnosis was fast because the profiler analysis script (Python over the JSON) made it trivial to surface `props=['ref']` in changeDescriptions
- ROADMAP and STATE updated incrementally, never let drift get worse than 1 commit

## What I'd do differently next time

- **Spot-check ROADMAP SCs against the code at phase-start** (would have caught the "200-300ms debounce" outdated target before drafting P19's CONTEXT)
- **Prefix profiler exports by phase from day one** (the `p18_*` filename now contains P19 data; future captures should use a fresh prefix)
- **Manual smoke checkpoint between major refactors** — the user did this implicitly via "go on" / "continue" but a more explicit "smoke this before next phase?" prompt would have caught the ref-callback issue before re-measuring
- **Document compiler-skip implication once at milestone level** (not in every phase context) — would shorten CONTEXT files
