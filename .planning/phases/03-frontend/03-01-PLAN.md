# Plan 03-01: Replace Recurrence Fields Across Types, Schema, Payload, and UI

## Goal

Remove all traces of `repetitions`, `end_date`, `recurrenceEndDateMode`, `recurrenceEndDate`, and `recurrenceRepetitions` from the frontend codebase. Replace with `current_installment`/`total_installments` in TypeScript types, `recurrenceCurrentInstallment`/`recurrenceTotalInstallments` in form schema, and two side-by-side NumberInputs in the UI component.

Purpose: Align the frontend with the backend API contract established in Phases 1-2, where `RecurrenceSettings` now uses `current_installment` and `total_installments`.

Output: All six FE requirements (FE-01 through FE-06) satisfied; `npm run build` passes.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| form input -> API payload | User-controlled numeric values sent to backend |
| form input -> UI render | User-controlled values rendered in error messages |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | NumberInput min bounds | mitigate | Zod schema enforces `z.number().int().nullable()` with `min={1}` on NumberInput; backend also validates >= 1 |
| T-03-02 | Tampering | current > total bypass | mitigate | `applySharedRefinements` rejects `current > total` before payload is built; backend double-validates |
| T-03-03 | Information Disclosure | Error message injection | accept | Error messages are static Portuguese strings, not user-controlled values; no XSS vector |
</threat_model>
