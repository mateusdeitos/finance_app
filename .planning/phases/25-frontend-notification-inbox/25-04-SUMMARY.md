# Phase 25 Plan 04: Inbox UI Surfaces Summary

Delivered the full notification inbox UI (Wave 3): the shared `NotificationInboxContent` list renderer (list/loading/empty/error/load-more/mark-all), the `NotificationRow` component, the mobile `NotificationInboxDrawer`, the desktop `NotificationInboxPage`, and the thin `/notifications` route. All four INBOX requirements (INBOX-01..04) are observable in the UI.

## What Was Built

### Task 1 — NotificationRow

`frontend/src/components/notifications/NotificationRow.tsx`:
- `UnstyledButton` wrapper (min-height 48px, `data-testid={NotificationsTestIds.Row(id)}`)
- Per-type `ThemeIcon variant="light" radius="md" size={36}`: charge_received=blue, charge_accepted=teal, split_created=violet, split_updated=orange; grey fallback for unknown types
- 8×8 `Box bg="blue.6" radius="50%"` unread dot or transparent spacer; `data-testid={NotificationsTestIds.UnreadDot(id)}`, `aria-hidden`
- `.unread` CSS module class: `background-color: light-dark(var(--mantine-color-blue-0), var(--mantine-color-dark-5))` — applied when unread
- Description via `describeNotification(n, ctx)` (Plan 02 pure function) with `fw={500} lineClamp={2}`, `c="dimmed"` when read
- Relative timestamp: `Intl.RelativeTimeFormat('pt-BR')` for ≤7 days, `Intl.DateTimeFormat('pt-BR', {dateStyle:'short'})` for older
- Trailing `IconChevronRight size={16}` dimmed
- Tap: `markRead(id)` (background, parent-owned mutation) + `router.navigate(deriveDeepLink({type, entity_type}))` + `onAfterTap?.()` for drawer close
- `aria-label` includes read state suffix `" — não lida"` when unread

### Task 2 — NotificationInboxContent (shared)

`frontend/src/components/notifications/NotificationInboxContent.tsx`:
- Owns `useNotificationInbox`, `useNotificationUnreadCount`, `useMarkNotificationRead`, `useMarkAllNotificationsRead`; `invalidateInbox` passed as `onSuccess` to both mutations
- Derives `partnerName` from `useMe + useAccounts` (accepted user_connection, picks the other user's name) — zero extra API calls
- Flattens cursor pages: `query.data?.pages.flatMap(p => p.notifications) ?? []`
- Batch-resolves amounts: `useResolveNotificationAmounts(notifications)` (≤2 requests/page)
- Loading: 3 `Skeleton height={56} radius="md"` rows
- Error: `data-testid={NotificationsTestIds.ErrorState}`, "Não foi possível carregar as notificações." + BtnRetry
- Empty: `data-testid={NotificationsTestIds.EmptyState}`, BellOff ThemeIcon + copy
- Populated: `NotificationRow` list with `Divider ml={56}` separators
- Mark-all button: `data-testid={NotificationsTestIds.BtnMarkAllRead}`, shown only when `unreadCount > 0`; hides optimistically after mark-all
- Load-more: `data-testid={NotificationsTestIds.BtnLoadMore}`, shown when `query.hasNextPage`; "Você está em dia" when no more pages
- Error toasts: fired at caller level via `mutate(id, { onError })` — no toast inside the mutation hooks

### Task 3 — Drawer + Page + Route

`frontend/src/components/notifications/NotificationInboxDrawer.tsx`:
- `useDrawerContext<void>()` — `opened/close/reject`
- `ResponsiveDrawer position="bottom" styles={{ content: { height: 'auto', maxHeight: '92dvh' } }}`
- Title "Notificações" `fw={700}` via `Title order={3} size="lg"`
- Close button: `aria-label="Fechar"`, `data-testid={NotificationsTestIds.DrawerClose}`
- `data-testid={NotificationsTestIds.Drawer}` on the drawer root
- `onRowTap={() => close(undefined as void)}` so a row tap closes the sheet
- Exports `openNotificationInboxDrawer()` — the Wave 4 open helper

`frontend/src/pages/NotificationInboxPage.tsx`:
- `Stack gap="md" p="md"` → `Title order={2} fw={700} size="lg"` → `NotificationInboxContent`
- No `Paper` wrapper, no `onRowTap` — page stays open on row tap

`frontend/src/routes/_authenticated.notifications.tsx`:
- Thin: `createFileRoute('/_authenticated/notifications')({ component: NotificationInboxPage })` only

## Deviations from Plan

**None.** Plan executed exactly as written. One minor process note: `routeTree.gen.ts` required manual pre-population of the `/_authenticated/notifications` entry so that `tsc -b` (which runs before vite) could resolve the route path type. The vite build plugin then regenerates the file correctly. This is a known bootstrap issue with TanStack Router's file-based code generation and `tsc` ordering.

## Known Stubs

None. All components are wired to real hooks and data:
- `NotificationInboxContent` calls live hooks from Plans 02/03
- `partnerName` derived from real `useAccounts` cached connection data
- `router.navigate` calls the real global router
- `deriveDeepLink` is the Phase 24 source of truth helper

## Threat Surface Scan

No new server surface. T-25-11 (open redirect): `deriveDeepLink` returns only fixed internal paths — `router.navigate({ to })` receives only `/charges`, `/transactions`, or `/` (guarded by `if (to !== '/')` in row tap handler). T-25-12 (XSS): `describeNotification` output is plain text rendered in Mantine `<Text>` — no `dangerouslySetInnerHTML`. T-25-13: no additional IDOR surface (data is IDOR-scoped server-side from Phase 23).
