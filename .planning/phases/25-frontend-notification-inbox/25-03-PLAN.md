<objective>
Build the D-25-1 batched amount resolver and the D-25-2 optimistic mark-read / mark-all-read mutations. This is the data-layer correctness core of the inbox.

Purpose: Resolve each notification's entity amount in at most 2 requests per page (one batched charges query via the existing List `id[]` filter, one batched transactions query via the new `/by-ids` handler from Plan 01) — NOT N+1 per row (the explicitly-rejected approach). Expose `{ amount, amountState: 'known'|'loading'|'missing' }` per notification, feeding describeNotification (Plan 02). Provide optimistic mark-read mutations whose `onMutate`/`onError` cache logic and `onSettled` unread-count reconcile live IN the hook (D-25-2), while the inbox-list invalidation is exposed as a caller `onSuccess?` option (CLAUDE.md "caller invalidates").

Output: `fetchChargesByIds`/`fetchTransactionsByIds` api fns; `useResolveNotificationAmounts` (plural, page-level) + test; `useMarkNotificationRead` + test; `useMarkAllNotificationsRead`.
</objective>
