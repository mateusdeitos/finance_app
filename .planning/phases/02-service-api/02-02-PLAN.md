# Plan 02-02: Regenerate Swagger Docs for New RecurrenceSettings Shape

## Goal
Regenerate Swagger/OpenAPI documentation so the spec reflects the new `RecurrenceSettings` struct with `current_installment` and `total_installments` fields. The old `repetitions` and `end_date` fields must not appear in the generated docs.

Purpose: The domain struct was updated in Phase 1, but `just generate-docs` was never run, so the swagger files still show the old fields. API consumers (the frontend) need accurate docs.

Output: Regenerated `docs/docs.go`, `docs/swagger.json`, `docs/swagger.yaml`.

<threat_model>
## Threats
- [LOW] Stale API documentation could mislead frontend developers: Mitigated by regenerating docs from current struct definitions.
</threat_model>

**Step 1: Verify handler annotations are correct (API-01)**

The handler annotations at `backend/internal/handler/transaction_handler.go` use:
- `@Param transaction body domain.TransactionCreateRequest true "Transaction data"` (line 32)
- `@Param transaction body domain.TransactionUpdateRequest true "Fields to update"` (line 62)

These reference the domain structs directly. Since the domain structs were already updated in Phase 1 to use `CurrentInstallment int json:"current_installment"` and `TotalInstallments int json:"total_installments"`, the annotations are already correct. No handler file changes needed.

If the handler annotations do NOT reference `domain.TransactionCreateRequest` / `domain.TransactionUpdateRequest` (e.g., they use inline structs or a separate request type), update them to reference the domain types.

**Step 2: Regenerate Swagger docs (API-02)**

Run from the `backend/` directory:

```bash
cd /workspace/backend && swag init -g cmd/server/main.go --output docs --parseInternal
```

This is equivalent to `just generate-docs`. It will parse all handler annotations and domain struct json tags, producing updated `docs/docs.go`, `docs/swagger.json`, and `docs/swagger.yaml`.

**Step 3: Verify the generated docs**

After generation, confirm:
- `swagger.json` contains `"current_installment"` and `"total_installments"` in the RecurrenceSettings schema
- `swagger.json` does NOT contain `"repetitions"` in the RecurrenceSettings schema
- `swagger.yaml` contains `current_installment` and `total_installments`

Note: `end_date` may appear in TransactionFilter (a query filter for date range) -- that is unrelated to RecurrenceSettings and is acceptable.
