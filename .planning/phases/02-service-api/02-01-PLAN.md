# Plan 02-01: Fix Create Loop to Honor CurrentInstallment

## Goal
Modify the transaction create loop in `createTransactions` so that when `CurrentInstallment > 1`, only installments from `CurrentInstallment` through `TotalInstallments` are created. Each installment must carry the correct `InstallmentNumber` (its full-series position) and the correct date offset (relative to `CurrentInstallment` as the base).

Purpose: This is the core behavioral change -- without it, `current_installment=3, total_installments=10` would still create 10 transactions starting from installment 1, defeating the entire feature purpose.

Output: Modified `transaction_create.go` with correct loop logic.

<threat_model>
## Threats
- [LOW] Integer overflow in loop bounds: CurrentInstallment and TotalInstallments are validated to be <= 1000 and >= 1 by validateRecurrenceSettings, so loop bounds are safe.
- [LOW] Off-by-one in date calculation: The date offset formula `(i - CurrentInstallment)` produces 0 for the first created installment (correct -- it gets the provided date). Verified by acceptance criteria.
</threat_model>

<action>
In `backend/internal/service/transaction_create.go`, function `createTransactions`, change the recurrence loop (currently at line 184) from:

```go
for i := 1; i <= recurrence.Installments; i++ {
```

to:

```go
for i := req.RecurrenceSettings.CurrentInstallment; i <= recurrence.Installments; i++ {
```

This changes the loop start from 1 to `CurrentInstallment`. Since `recurrence.Installments` already equals `TotalInstallments` (set by `RecurrenceFromSettings` in Phase 1), the loop end is already correct.

Next, change the date offset calculation inside the loop. Currently (line 189):

```go
Date: s.incrementInstallmentDate(req.Date, lo.FromPtr(req.RecurrenceSettings).Type, i-1),
```

Change to:

```go
Date: s.incrementInstallmentDate(req.Date, lo.FromPtr(req.RecurrenceSettings).Type, i-req.RecurrenceSettings.CurrentInstallment),
```

This ensures:
- Installment `CurrentInstallment` (the first created) gets offset 0 (the provided base date)
- Installment `CurrentInstallment + 1` gets offset 1 (base_date + 1 interval)
- etc.

No other changes needed:
- `InstallmentNumber` is already set to `&i` at line 188 -- with the new loop start, `i` will be 3, 4, ..., 10 for current=3/total=10. This is correct.
- `injectLinkedTransactions` already copies `transaction.InstallmentNumber` to linked transactions (lines 370 and 414), so splits and transfers inherit the correct number.
- `RecurrenceFromSettings` already stores `TotalInstallments` as `Installments` (done in Phase 1).
- `validateRecurrenceSettings` is already shared between create and update (confirmed by grep -- called from both `transaction_create.go:86` and `transaction_update.go:778,964`).

Per CRE-04: RecurrenceFromSettings already uses TotalInstallments -- no change needed (verified in Phase 1).
Per CRE-05: LinkedTransactions already inherit InstallmentNumber -- no change needed.
Per UPD-01: Update already calls validateRecurrenceSettings -- no change needed (verified in Phase 1).
</action>
