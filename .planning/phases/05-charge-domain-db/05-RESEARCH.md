## Summary

Phase 5 is entirely structural: it creates the Charge data model and persists it in the DB. There are no service or handler concerns in scope. The work divides into four orthogonal deliverables: (1) the `ChargeStatus` enum + `Charge` domain struct with `ValidateTransition`, (2) the `entity.Charge` GORM struct with `BeforeCreate`/`BeforeUpdate`/`ToDomain`/`FromDomain`, (3) two Goose SQL migration files, and (4) the `ChargeID *int` field addition to `domain.Transaction` and `entity.Transaction`.

All patterns needed are already present in the codebase. The `UserConnectionStatusEnum` pattern in `domain/user_connection.go` is the direct template for `ChargeStatus`. The `entity.Settlement` pattern (with `BeforeCreate`, `BeforeUpdate`, `ToDomain`, `FromDomain`) is the direct template for `entity.Charge`. The `entity.Transaction.CategoryID *int` field is the direct template for `ChargeID *int` ÔÇö GORM handles `*int` nullable FK columns natively without extra tags.

**Primary recommendation:** Follow existing patterns exactly. No new libraries, no new patterns, no hand-rolled behavior.

## Architecture Patterns

### Pattern 1: Typed String Enum in Domain Layer

**Template:** `internal/domain/user_connection.go`

```go
// Source: backend/internal/domain/user_connection.go
type UserConnectionStatusEnum string

const (
    UserConnectionStatusPending  UserConnectionStatusEnum = "pending"
    UserConnectionStatusAccepted UserConnectionStatusEnum = "accepted"
    UserConnectionStatusRejected UserConnectionStatusEnum = "rejected"
)

func (s UserConnectionStatusEnum) IsValid() bool {
    return s == UserConnectionStatusPending || s == UserConnectionStatusAccepted || s == UserConnectionStatusRejected
}
```

Apply this pattern for `ChargeStatus`:

```go
type ChargeStatus string

const (
    ChargeStatusPending   ChargeStatus = "pending"
    ChargeStatusPaid      ChargeStatus = "paid"
    ChargeStatusRejected  ChargeStatus = "rejected"
    ChargeStatusCancelled ChargeStatus = "cancelled"
)

func (s ChargeStatus) IsValid() bool {
    return s == ChargeStatusPending || s == ChargeStatusPaid ||
        s == ChargeStatusRejected || s == ChargeStatusCancelled
}
```

### Pattern 2: ValidateTransition in Domain Struct

`ChargeStatus` has a state machine with terminal states. The transition guard lives on the domain struct (not the service), matching the principle that domain types own their business rules.

```go
// ErrInvalidStatusTransition is a domain-level sentinel error
var ErrInvalidStatusTransition = errors.New("invalid charge status transition")

func (c *Charge) ValidateTransition(newStatus ChargeStatus) error {
    switch c.Status {
    case ChargeStatusPending:
        if newStatus == ChargeStatusPaid || newStatus == ChargeStatusRejected || newStatus == ChargeStatusCancelled {
            return nil
        }
    }
    return ErrInvalidStatusTransition
}
```

Note: `ErrInvalidStatusTransition` is a plain `errors.New` value in `internal/domain/charge.go`. The service layer (Phase 6) will wrap it in a `pkg/errors.ServiceError` before returning to handlers.

### Pattern 3: GORM Entity with Hooks and Conversion

**Template:** `internal/entity/settlement.go` and `internal/entity/user_connection.go`

```go
// Source: backend/internal/entity/settlement.go
type Settlement struct {
    ID    int    `gorm:"primaryKey;autoIncrement"`
    ...   fields with gorm tags on required/special ones ...
}

func (Settlement) BeforeCreate(tx *gorm.DB) error {
    now := time.Now()
    tx.Statement.SetColumn("created_at", now)
    tx.Statement.SetColumn("updated_at", now)
    return nil
}

func (s *Settlement) BeforeUpdate(tx *gorm.DB) error {
    tx.Statement.SetColumn("updated_at", time.Now())
    return nil
}

func (s *Settlement) ToDomain() *domain.Settlement { ... }
func SettlementFromDomain(d *domain.Settlement) *Settlement { ... }
```

Key observations from the codebase:
- `ID int` with `gorm:"primaryKey;autoIncrement"` ÔÇö required tag for the PK
- Required non-nullable fields use `gorm:"not null"` tag on `Settlement` but **not** on `Transaction` or `Category` ÔÇö the project is inconsistent; for `Charge` entity, follow `Settlement` (be explicit with `not null` on required fields)
- `*int` nullable FK fields (`CategoryID *int`, `ParentID *int`, `OriginalUserID *int` in existing entities) use **no special GORM tag** ÔÇö GORM infers nullable from the Go pointer type [VERIFIED: entity/transaction.go, entity/category.go]
- `*time.Time` fields for `CreatedAt`/`UpdatedAt` ÔÇö no GORM tag needed; hooks set them [VERIFIED: all entities]

### Pattern 5: Goose Migration Format

**Template:** `migrations/20260309000000_create_settlements_table.sql` (CREATE TABLE) and `migrations/20260325000001_add_emoji_to_categories.sql` (ALTER TABLE)

Two distinct patterns:

**CREATE TABLE (simple ÔÇö no `StatementBegin` wrapper needed):**
```sql
-- +goose Up
CREATE TABLE ...;
CREATE INDEX ...;

-- +goose Down
DROP TABLE IF EXISTS ...;
DROP TYPE IF EXISTS ...;
```

**ALTER TABLE (requires `StatementBegin`/`StatementEnd` wrapper):**
```sql
-- +goose Up
-- +goose StatementBegin
ALTER TABLE transactions ADD COLUMN charge_id INT REFERENCES charges(id) ON DELETE SET NULL;
CREATE INDEX idx_transactions_charge_id ON transactions(charge_id) WHERE charge_id IS NOT NULL;
-- +goose StatementEnd

-- +goose Down
-- +goose StatementBegin
DROP INDEX IF EXISTS idx_transactions_charge_id;
ALTER TABLE transactions DROP COLUMN charge_id;
-- +goose StatementEnd
```

**Key observation:** The `create_settlements_table.sql` migration does NOT use `StatementBegin`/`StatementEnd` wrappers for `CREATE TABLE`. Only the `ALTER TABLE` migrations use them. The CONTEXT.md schema for charges uses plain `CREATE TABLE` syntax ÔÇö follow the `create_settlements_table.sql` format exactly. [VERIFIED: migrations/20260309000000_create_settlements_table.sql]

**Timestamp type:** All existing tables use `TIMESTAMPTZ` (following the 2026-01-09 migration that converted all columns). New `charges` table must use `TIMESTAMPTZ` for `created_at` and `updated_at`. [VERIFIED: migrations/20260109123226_timestamp_with_timezone.sql]

**CONTEXT.md schema uses `TIMESTAMPTZ`** ÔÇö already correct.

**Status column type:** The CONTEXT.md schema uses `VARCHAR(20) NOT NULL DEFAULT 'pending'` for status. The `user_connections` table uses a `CREATE TYPE connection_status AS ENUM (...)` instead. Both approaches work. The CONTEXT.md decision is to use `VARCHAR(20)` ÔÇö follow that exactly (it avoids needing `DROP TYPE` in rollback).

### Anti-Patterns to Avoid

- **Do not use `database/sql.NullInt64` for nullable FKs** ÔÇö the project consistently uses `*int` pointer type. GORM handles this natively. [VERIFIED: entity/transaction.go CategoryID, entity/category.go ParentID]
- **Do not create a DB ENUM type for `charge_status`** ÔÇö CONTEXT.md specifies `VARCHAR(20)`, avoiding the complexity of `DROP TYPE` in rollbacks and migration ordering issues.
- **Do not add `gorm:"column:charge_id"` tag unless column name differs from snake_case of field name** ÔÇö GORM derives `charge_id` from `ChargeID` automatically.
- **Do not skip `-- +goose Down` section** ÔÇö all existing migrations include rollback logic.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Timestamp auto-set on create/update | Manual `time.Now()` assignment in service | `BeforeCreate`/`BeforeUpdate` GORM hooks | Consistent with all existing entities; hooks fire on all write paths |
| Nullable FK column | `database/sql.NullInt64` | `*int` Go pointer | GORM natively maps pointer to nullable column; existing entities confirm this pattern |
| Status validation | Ad-hoc string comparison in service | `IsValid()` method on `ChargeStatus` type + `ValidateTransition` on `Charge` | Keeps invariants in domain layer, not scattered across services |

## Common Pitfalls

### Pitfall 1: Timestamp Column Type Mismatch

**What goes wrong:** Writing `TIMESTAMP` instead of `TIMESTAMPTZ` for `created_at`/`updated_at` in the charges migration.
**Why it happens:** The `create_settlements_table.sql` (the template) uses plain `TIMESTAMP` ÔÇö it predates the 2026-01-09 timezone migration that converted all existing tables.
**How to avoid:** All NEW tables must use `TIMESTAMPTZ` from the start. [VERIFIED: migrations/20260109123226_timestamp_with_timezone.sql]
**Warning signs:** If created_at appears without timezone offset in psql `\d charges`.

### Pitfall 2: Forgetting `gorm:"primaryKey;autoIncrement"` Tag on Entity

**What goes wrong:** GORM may not correctly identify the PK, causing insert failures or duplicate inserts.
**Why it happens:** `entity.Transaction` omits the tag on `ID` (relies on GORM convention), but `entity.Settlement` and `entity.TransactionRecurrence` use it explicitly.
**How to avoid:** Follow `entity.Settlement` pattern ÔÇö explicitly tag `ID int \`gorm:"primaryKey;autoIncrement"\``.

### Pitfall 3: `ON DELETE` Behavior on charge_id FK

**What goes wrong:** Setting `charge_id` FK to `ON DELETE CASCADE` instead of `ON DELETE SET NULL`.
**Why it happens:** `CASCADE` is a common default; `SET NULL` is the correct behavior here (transactions outlive charges).
**How to avoid:** The CONTEXT.md schema is explicit: `REFERENCES charges(id) ON DELETE SET NULL`. Follow it exactly.

### Pitfall 4: Migration Ordering ÔÇö charges table must exist before transactions FK

**What goes wrong:** Running `add_charge_id_to_transactions.sql` before `create_charges_table.sql`.
**Why it happens:** Goose applies migrations in filename-sorted order. The timestamp `20260414000001` is after `20260414000000`.
**How to avoid:** Use `000000` for the charges table and `000001` for the transactions ALTER ÔÇö exactly as specified in CONTEXT.md. [VERIFIED: CONTEXT.md decision 4]

### Pitfall 5: ValidateTransition Does Not Return ServiceError

**What goes wrong:** Returning a `pkg/errors.ServiceError` from `ValidateTransition` in the domain layer.
**Why it happens:** The domain layer is supposed to be infrastructure-free; `pkg/errors` imports `github.com/labstack/echo/v4` (HTTP library).
**How to avoid:** Return a plain `error` from `ValidateTransition` (e.g., `errors.New("invalid charge status transition")`). The service layer (Phase 6) wraps it in `ServiceError`.
**Confirmed:** `pkg/errors/errors.go` imports `echo` ÔÇö domain packages must not import it. [VERIFIED: pkg/errors/errors.go line 8]

```sql
-- Source: pattern from migrations/20260309000000_create_settlements_table.sql
-- File: backend/migrations/20260414000000_create_charges_table.sql
-- +goose Up
CREATE TABLE charges (
    id                  SERIAL PRIMARY KEY,
    charger_user_id     INT NOT NULL REFERENCES users(id),
    payer_user_id       INT NOT NULL REFERENCES users(id),
    charger_account_id  INT REFERENCES accounts(id),
    payer_account_id    INT REFERENCES accounts(id),
    connection_id       INT NOT NULL REFERENCES user_connections(id) ON DELETE RESTRICT,
    period_month        INT NOT NULL CHECK (period_month BETWEEN 1 AND 12),
    period_year         INT NOT NULL,
    description         TEXT,
    status              VARCHAR(20) NOT NULL DEFAULT 'pending',
    created_at          TIMESTAMPTZ,
    updated_at          TIMESTAMPTZ
);

CREATE INDEX idx_charges_charger_user_id ON charges(charger_user_id);
CREATE INDEX idx_charges_payer_user_id ON charges(payer_user_id);
CREATE INDEX idx_charges_connection_id ON charges(connection_id);
CREATE INDEX idx_charges_status ON charges(status);

-- +goose Down
DROP TABLE IF EXISTS charges;
```

### Known Threat Patterns for this stack

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Invalid status transition (e.g., re-paying a paid charge) | Tampering | `ValidateTransition` enforces state machine in domain layer |
