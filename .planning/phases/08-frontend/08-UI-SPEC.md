## Spacing Scale

Mantine's internal spacing tokens are used throughout the existing codebase. All spacing in new components must use Mantine's named tokens, not raw pixel values.

| Token | Mantine Prop | Approximate px | Usage |
|-------|-------------|----------------|-------|
| xs | `gap="xs"` | 4px | Icon gaps, badge-to-text spacing, inline tight groups |
| sm | `gap="sm"` / `p="xs"` | 8px | Card internal padding, compact button groups |
| md | `gap="md"` / `p="md"` | 16px | Default element spacing, drawer body padding |
| lg | `gap="lg"` / `p="lg"` | 24px | Section breaks within page |
| xl | `gap="xl"` | 32px | Layout gaps between major page sections |

Exceptions:
- Sidebar badge uses inline positioning relative to NavLink icon — no explicit spacing token; badge overlays icon corner via `pos="absolute"` or Mantine `Indicator` component
- Mobile bottom bar safe area: `pb="5rem"` on page Stack (matches existing transactions page pattern)
- Touch targets for action buttons on charge cards: minimum 44px height (Mantine default Button md size satisfies this)

Source: existing usage in `_authenticated.transactions.tsx`, `TransactionRow.tsx`

## Typography

All typography uses Mantine's `Text` and `Title` components with the following size/weight constraints. These match the patterns established in `TransactionRow.tsx` and `AppLayout.tsx`.

Two weights only: 400 (regular) for body and metadata; 700 (bold) for emphasized values and headings.

| Role | Mantine Size | Weight | Line Height | Usage |
|------|-------------|--------|-------------|-------|
| Body | `size="sm"` (14px) | 400 regular | 1.5 | Charge description, partner name, period label |
| Label | `size="xs"` (12px) | 400 regular | 1.4 | Secondary metadata, date labels, status badge text |
| Heading | `size="md"` (16px) | 700 bold (`fw={700}`) | 1.4 | Amount display on ChargeCard |
| Display | `size="lg"` (18px) | 700 bold (`fw={700}`) | 1.2 | Page heading, drawer title, balance preview amount |

Constraints:
- Do NOT introduce new font sizes. Use only the four sizes declared above.
- Do NOT introduce new weights. Use only 400 and 700.
- Partner name on ChargeCard: `size="md" fw={400}` — the larger size provides sufficient hierarchy over `size="sm"` metadata without a separate weight.
- Amount values: `fw={700}` — aligns with the Display tier and creates a clear two-tier weight system: regular text vs. emphasized values.
- Input labels use Mantine's built-in label styling — do not override.

Source: `AppLayout.tsx` line 57 (`fw={700} size="lg"`); weight reduction from 4→2 is a UI-SPEC decision (checker fix 2026-04-16).

## Color

Mantine semantic color tokens are used exclusively. Raw hex values are not used in component code.

| Role | Mantine Token | Usage |
|------|--------------|-------|
| Dominant (60%) | `var(--mantine-color-body)` | Page background, drawer background, card background |
| Secondary (30%) | `var(--mantine-color-default)` / `c="dimmed"` | Card surfaces, metadata text, secondary labels |
| Accent (10%) | `blue.7` (`#457b9d`) | Primary action buttons, active nav link, accept action button |
| Success/Credit | `teal` / `green.8` | Positive balance preview ("Devem a você"), paid status badge |
| Destructive | `red` / `red.6` (`#e63946`) | Reject and cancel action buttons, reject confirmation, destructive badge, sidebar pending-count badge |
| Warning/Pending | `yellow.6` (`#ef8019`) | Pending status badge background |

Accent reserved for: primary CTA button ("Nova Cobrança"), active NavLink highlight, accept action button.

Destructive (`red`) reserved for: reject button, cancel button, reject/cancel confirmation dialogs, sidebar pending-count badge (standard notification convention — red signals required user action).

Status badge color mapping (Mantine `Badge` component):
- `pending` → `color="yellow"` variant `light`
- `paid` → `color="teal"` variant `light`
- `rejected` → `color="red"` variant `light`
- `cancelled` → `color="gray"` variant `light`

Source: `theme.ts` (color palette), `TransactionRow.tsx` line 238 (credit/debit teal/red pattern); sidebar badge color corrected to red (checker fix 2026-04-16 — red is the standard notification convention).

## Focal Point

**Primary screen focal point: the Charges listing page.**

The visual hierarchy anchors on the two-tab strip ("Recebidas" / "Enviadas") immediately below the sticky header. Within each tab, ChargeCards with pending status are the dominant visual targets: the yellow "Pendente" badge draws the eye first, the bold amount (`fw={700}`) is the second anchor, and the teal "Aceitar" / red "Recusar" action buttons are the call-to-action terminus. The "Nova Cobrança" button in the sticky header is the page-level primary action and uses accent blue to stay distinct from card-level actions.

Single-screen hierarchy order:
1. Sticky header — "Nova Cobrança" CTA (accent blue, top-right)
2. Tabs strip — orientation anchor (received vs. sent)
3. ChargeCard amount (`fw={700}`) — value prominence
4. ChargeCard status badge (color-coded) — state recognition
5. Action buttons (teal / red) — task completion

## Layout Contracts

### Charges Page (`_authenticated.charges.tsx`)

Structure mirrors `_authenticated.transactions.tsx`:

```
Stack gap="md"
  ├── sticky header Box (same sticky pattern as transactions page)
  │     ├── Group justify="space-between"
  │     │     ├── PeriodNavigator (reused, adapted for /charges route)
  │     │     └── Button "Nova Cobrança" (leftSection=<IconPlus>)
  │     └── [no filters — D-04: no status filter]
  └── Tabs (defaultValue="received")
        ├── Tabs.List
        │     ├── Tabs.Tab value="received" "Recebidas"
        │     └── Tabs.Tab value="sent"     "Enviadas"
```

### ChargeCard

```
Card withBorder radius="md" p="md"
  ├── Group justify="space-between" align="flex-start"
  │     ├── Stack gap={2}
  │     │     ├── Text size="md" fw={400}  — partner name
  │     │     ├── Text size="sm" c="dimmed" — period label "MM/YYYY"
  │     │     └── Text size="sm" c="dimmed" lineClamp={1} — description (if present)
  │     └── ChargeStatusBadge (status)
  └── Group justify="space-between" align="center" mt="sm"
        ├── Text size="md" fw={700} — amount (from balance preview or charge amount)

```

Action buttons per role/status:
- Received + pending: `Button size="xs" color="teal"` "Aceitar" + `Button size="xs" color="red" variant="light"` "Recusar"
- Sent + pending: `Button size="xs" color="red" variant="light"` "Cancelar"
- Any terminal status (paid/rejected/cancelled): no action buttons

### Sidebar Badge (AppLayout.tsx)

Add to `navLinks` array:
```tsx
{
  label: "Cobranças",
  icon: IconCreditCard,    // IconCreditCard from @tabler/icons-react
  to: "/charges",
  badge: pendingCount      // number | undefined; badge hidden when 0 or undefined
}
```

NavLink render with badge:
```tsx
<NavLink
  key={to}
  component={Link}
  to={to}
  label={label}
  leftSection={<Icon size={18} />}
  rightSection={badge ? <Badge size="xs" circle color="red">{badge}</Badge> : undefined}
  active={currentPath === to}
  onClick={close}
/>
```

Badge uses `color="red"` to signal required user action (standard notification convention). Disappears when `pendingCount === 0`.

## Interaction Contracts

### Mutation UX Pattern (all charge actions — D-09/D-10/D-11/D-12)

All four mutations follow this identical pattern:

1. Submit button: `loading={mutation.isPending}` + `disabled={mutation.isPending}`
2. On success: close drawer/modal + call `invalidate()` for `QueryKeys.Charges` AND `QueryKeys.ChargesPendingCount` + show Mantine notification
3. On error: translate error tags via `mapTagsToFieldErrors` extended with `CHARGE.*` tags; display as field errors or `_general` alert

Success notification shape:
```tsx
notifications.show({
  color: "teal",
  title: "{Action} realizada",
  message: "{Confirmation message}",
  autoClose: 3000,
})
```

Error notification shape (for non-field errors):
```tsx
notifications.show({
  color: "red",
  title: "Erro",
  message: "{Translated error message}",
  autoClose: 5000,
})
```

### Non-Optimistic Mutations (from STATE.md)

No optimistic updates. All charge state transitions wait for server confirmation before updating UI. On success, invalidate queries to refetch from server.

### Period Navigation

`PeriodNavigator` is reused but must be adapted for the `/charges` route (it currently hard-codes `from: '/transactions'` and `from: '/_authenticated/transactions'`). The charges page version reads from its own route search params.

### Loading States

- Charges list: Mantine `Skeleton` component — 3 skeleton cards while `query.isLoading`
- Pending count: badge renders nothing (hidden) while loading
- Balance preview in drawers: `Skeleton height={40}` while balance query is pending

### Empty States

Received tab empty:
- Heading: "Nenhuma cobrança recebida"
- Body: "Você não tem cobranças recebidas neste período."

Sent tab empty:
- Heading: "Nenhuma cobrança enviada"
- Body: `Stack align="center" gap="sm"` + Text + Button "Nova Cobrança" (shortcut)

## Error Tag Mappings (extend apiErrors.ts)

New entries for `tagToField` in `mapTagsToFieldErrors`:

```ts
'CHARGE.INVALID_CONNECTION_ID': 'connection_id',
'CHARGE.CONNECTION_NOT_ACCEPTED': 'connection_id',
'CHARGE.INVALID_ACCOUNT_ID': 'my_account_id',
'CHARGE.INVALID_PAYER': '_general',
'CHARGE.CHARGE_NOT_PENDING': '_general',
'CHARGE.AMOUNT_MUST_BE_POSITIVE': 'amount',
```

General errors (no tag match) display as `_general` alert inside the drawer/modal. Portuguese message from the `message` field of the API error response.
