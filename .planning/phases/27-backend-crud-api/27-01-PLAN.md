    `Create` — race-safe capped conditional INSERT (SECURITY T-27-02, no exact analog; compose from ConditionalAccept). Marshal the payload via the entity's jsonb wrapper, run a single parameterized `INSERT ... SELECT ...    
    ```go
    func (r *transactionTemplateRepository) Create(ctx context.Context, t *domain.TransactionTemplate) (*domain.TransactionTemplate, error) {
        ent := entity.TransactionTemplateFromDomain(t)
        payloadJSON, err := ent.Payload.Value() // driver.Value ([]byte json)
        if err != nil {
            return nil, err
        }
        var created entity.TransactionTemplate
        // Single atomic statement: the COUNT subquery + INSERT evaluate together,
        // so two concurrent creates at count=2 cannot both land a 4th row.
        rows, err := GetTxFromContext(ctx, r.db).Raw(`
            INSERT INTO transaction_templates (user_id, name, payload, created_at, updated_at)
            SELECT ?, ?, ?, NOW(), NOW()
            WHERE (SELECT COUNT(*) FROM transaction_templates WHERE user_id = ?) < 3
            RETURNING id, user_id, name, payload, created_at, updated_at
        `, ent.UserID, ent.Name, payloadJSON, ent.UserID).Rows()
        if err != nil {
            return nil, err
        }
        defer rows.Close()
        if !rows.Next() {
            return nil, ErrTemplateLimitReached
        }
        if err := GetTxFromContext(ctx, r.db).ScanRows(rows, &created); err != nil {
            return nil, err
        }
        return created.ToDomain(), nil
    }
    ```
    (If `Raw(...).Rows()` + `ScanRows` proves awkward with the jsonb Scan, the executor may instead use `.Exec` for the conditional INSERT, check `RowsAffected == 0` → `ErrTemplateLimitReached`, then re-`First` by `(user_id, name)` to hydrate — either shape is acceptable so long as it stays a SINGLE conditional INSERT gated by the COUNT subquery, parameterized, no string interpolation of user input. Do NOT do a separate `SELECT COUNT` then a conditional `if` in Go — that reintroduces the TOCTOU race.)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| service→repository | repository receives `userID` already resolved from auth context; it must still scope every query by it (defense in depth) |
| repository→PostgreSQL | user-controlled `name`/`payload` cross into SQL; must be parameterized, never string-interpolated |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-27-01 | Information Disclosure / Elevation | GetByIDForUser/Update/Delete | mitigate | Every query filters `WHERE id = ? AND user_id = ?`; `RowsAffected == 0` / `ErrRecordNotFound` → `pkgErrors.NotFound` (404, never 403 — existence not leaked). Verified by grep gate (`user_id = ?` >= 4, `NotFound` >= 3). |
| T-27-02 | Elevation of Privilege (cap bypass / TOCTOU) | Create | mitigate | Single atomic conditional `INSERT ... SELECT ... WHERE (SELECT COUNT(*) ...) < 3` — the count and insert evaluate in one statement, so two concurrent creates at count=2 cannot both succeed. `RowsAffected == 0` → `ErrTemplateLimitReached`. Grep gate forbids a Go-side count-then-if. |
| T-27-03 | Tampering (SQL injection via name/payload) | Create/Update/List | mitigate | All SQL uses parameterized `?` bind vars; payload marshaled via the typed entity jsonb `Value()`. No `fmt.Sprintf` into SQL. |
| T-27-04 | Tampering (stray/malicious payload keys) | Create/Update DTO | mitigate | Request `Payload` binds into the strict `TransactionTemplatePayload`; unknown keys (`amount`/`date`/typos) dropped by the Go json decoder; canonical struct re-serialized on write (D-01/D-02). |
</threat_model>
