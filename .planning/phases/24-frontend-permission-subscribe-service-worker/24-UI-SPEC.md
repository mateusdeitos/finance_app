## Typography

Declared weights: **500** and **700** only. Body and caption inherit Mantine's framework default (400) and are not counted as declared weights.

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Body | 16px (1rem) | inherited (400) | 1.5 | Body text, form field labels, drawer content |
| Label | 14px (0.875rem) | 500 | 1.4 | Nav item labels, toggle labels, setting row descriptions |
| Caption | 12px (0.75rem) | inherited (400) | 1.4 | Helper text, timestamps, dimmed secondary strings |
| Heading | 18px (Mantine `size="lg"`) | 700 | 1.2 | Drawer titles (e.g. "Configurações"), brand wordmark, section heading emphasis |

Size scale: 12 / 14 / 16 / 18px.

Rules:
- `size="sm"` in Mantine renders ~14px; `size="xs"` renders ~12px; `size="lg"` renders ~18px; default is 16px.
- `fw={500}` is the standard label weight (see `MobileMoreDrawer`, `DesktopSidebar`).
- `fw={700}` is used for drawer titles, section headings, and the brand wordmark.
- `c="dimmed"` renders via `var(--mantine-color-dimmed)` — use for secondary helper text.
- Body copy min-size: 16px enforced by `index.css` to prevent iOS zoom on focus.

## Visible UI Surfaces in Phase 24

Phase 24 adds three visible UI areas on top of invisible plumbing (SW registration, push subscription API calls):

1. **Notification Toggle Control** — an in-app Switch row in the "Mais" / settings area (CTRL-02)
2. **Permission Enable Affordance** — the explicit user action that triggers the browser prompt (CTRL-01)
3. **Toast Feedback** — subscribe/unsubscribe success and failure notifications

### Surface 1 — Notification Toggle Row

#### Placement

**Mobile**: Add as a new `MoreItem` inside `MobileMoreDrawer` (`src/components/MobileMoreDrawer.tsx`), rendered with the same `UnstyledButton` + icon + label pattern used by "Criar Conexão" and "Importar transações". Placed above the "Sair" item.

**Desktop**: Add inside the `DesktopSidebar` user `Menu.Dropdown`, as a non-button `div` row (same pattern as the existing `themeMenuItem` row that holds `ThemeToggle`). Placed above the `Menu.Divider` before "Sair".

Layout of the toggle row (both surfaces):
```
[Icon 20px]  [Label text]  [Switch — right-aligned]
```
- Use Mantine `Group` with `justify="space-between"` and `align="center"`, `wrap="nowrap"`.
- Icon: `IconBell` (when enabled) / `IconBellOff` (when disabled) — size 20, stroke 1.8.
- Switch: Mantine `<Switch>` component, no label prop (label lives in the Group row), `size="md"`.
- Minimum row height: 44px (touch target).

#### State Machine

The toggle must represent exactly five states. The Switch `checked` prop, `disabled` prop, and accompanying text change per state:

| State | Switch `checked` | Switch `disabled` | Icon | Label | Helper text below label |
|-------|-----------------|-------------------|------|-------|------------------------|
| `default` (permission never asked, not subscribed) | `false` | `false` | `IconBellOff` | "Notificações" | "Toque para ativar" |
| `requesting` (browser prompt open / subscribe in flight) | `false` | `true` | `IconBell` | "Notificações" | "Aguardando permissão..." |
| `enabled` (subscribed and active) | `true` | `false` | `IconBell` | "Notificações" | "Ativadas neste dispositivo" |
| `denied` (browser permission = "denied") | `false` | `true` | `IconBellOff` | "Notificações" | "Bloqueadas — ative nas configurações do navegador" |
| `unsupported` (browser lacks Push API or SW) | `false` | `true` | `IconBellOff` | "Notificações" | "Não suportado neste navegador" |

Color rules per state:
- `enabled`: Switch color `"blue"` (Mantine default primary).
- `denied`: Icon and helper text use `c="red"` (Mantine red semantic).
- `unsupported`: Icon and helper text use `c="dimmed"`.
- `requesting`: Icon uses `c="dimmed"`; consider `Loader size="xs"` inline next to label instead of the icon when in this state (open decision — see below).
- `default`: Icon uses `c="dimmed"`.

#### Open Decision OD-1: Requesting-state icon vs loader

Recommended default: replace `IconBellOff` with a Mantine `Loader size="xs" color="blue"` next to "Aguardando permissão..." when `state === 'requesting'`. This gives unambiguous feedback that an async action is in progress. Executor should implement this default unless overridden.

### Surface 2 — Permission Enable Affordance (CTRL-01)

The browser `Notification.requestPermission()` call must ONLY fire when the user explicitly taps/clicks the toggle from `default` state to `requesting`. This is satisfied by wiring the Switch `onChange` handler to call `requestPermission()` + subscribe flow. No auto-prompt on mount. No banner or modal needed — the Switch itself is the affordance.

There is NO separate modal, banner, or onboarding step for this phase. The toggle IS the explicit action.

**Important**: When `Notification.permission === 'denied'`, the Switch must be disabled (non-interactive). No in-app modal can fix this — only the browser settings can. The helper text ("Bloqueadas — ative nas configurações do navegador") is the sole guidance. Do not add a "Como ativar" link or button.

---

### Surface 3 — Toast Feedback (subscribe / unsubscribe)

All feedback uses `@mantine/notifications` (`notifications.show()`), matching the established app pattern. Position: `top-right` (configured globally in `main.tsx`). `autoClose: 3000` for all non-loading toasts.

| Event | `color` | `title` | `message` |
|-------|---------|---------|-----------|
| Subscribe success | `"teal"` | `"Notificações ativadas"` | `"Você receberá notificações neste dispositivo."` |
| Unsubscribe success | `"teal"` | `"Notificações desativadas"` | `"Notificações desativadas neste dispositivo."` |
| Subscribe error (API failure) | `"red"` | `"Erro ao ativar"` | `"Não foi possível ativar as notificações. Tente novamente."` |
| Unsubscribe error (API failure) | `"red"` | `"Erro ao desativar"` | `"Não foi possível desativar as notificações."` |
| Permission denied by user | none (no toast) | — | — (toggle transitions to `denied` state, helper text is sufficient) |

Note: Do NOT show a toast when the user denies the browser permission prompt. The state change to `denied` with the helper text is the feedback mechanism.

### Surface 4 — OS/Browser Push Notification Content (SC4, CTRL-03)

The service worker's `push` event handler constructs an OS/browser notification. This is rendered by the OS — not by the app — but the title/body contract must be specified here so the executor knows what strings to use.

#### Title + Body Copy Contract

| Event type (backend) | `title` | `body` |
|----------------------|---------|--------|
| `charge_received` | `"Nova cobrança"` | `"{partner_name} criou uma cobrança para você."` |
| `charge_accepted` | `"Cobrança aceita"` | `"{partner_name} aceitou sua cobrança."` |
| `split_created` | `"Nova transação dividida"` | `"{partner_name} adicionou uma transação dividida."` |
| `split_updated` | `"Transação dividida atualizada"` | `"{partner_name} atualizou uma transação dividida."` |

`{partner_name}` is resolved from the push payload delivered by the backend. If the name is absent, fall back to `"Seu parceiro(a)"`.

#### Notification Options (passed to `self.registration.showNotification`)

```
icon: '/icon-192.png'
badge: '/icon-192.png'
data: { url: '<deep-link-url>' }  // used by notificationclick handler
tag: '<event-type>-<entity-id>'   // prevents duplicate stacking for the same entity
```

`tag` prevents duplicate OS notifications for the same entity if the backend fires twice. Pattern: `"charge-{id}"` or `"transaction-{id}"`.

#### Deep-link URL Contract (CTRL-03)

| Event type | Navigation target | URL pattern |
|------------|------------------|-------------|
| `charge_received` | Charges page | `/charges` |
| `charge_accepted` | Charges page | `/charges` |
| `split_created` | Transactions page | `/transactions` |
| `split_updated` | Transactions page | `/transactions` |

Phase 24 navigates to the list page, not the individual entity (no entity deep-link drawer open from SW click). Opening the exact entity from a notification tap is deferred until Phase 25 adds the inbox with full entity references, or a later enhancement. The SC5 criterion ("no extra tap or search is required") is satisfied by landing on the list page for the relevant entity type. The executor must not attempt to open a drawer from the service worker's `notificationclick` handler.

#### `notificationclick` handler behavior

```
event.waitUntil(
  clients.matchAll({ type: 'window', includeUncontrolled: true }).then(clientList => {
    // If app is already open: focus it and postMessage the URL
    // If app is not open: open the URL
  })
)
```

No UI visible in this handler — purely behavioral.

## Interaction Behavior

### Switch onChange flow

```
user taps Switch
  ├─ state === 'default' → call requestPermission()
  │     ├─ permission granted → call subscribe() → show requesting → on success: enabled + success toast
  │     │                                         → on error: default + error toast
  │     └─ permission denied → denied state (no toast)
  │
  ├─ state === 'enabled' → call unsubscribe()
  │     ├─ success → default state + success toast
  │     └─ error → stay enabled + error toast
  │
  ├─ state === 'requesting' → noop (Switch is disabled)
  ├─ state === 'denied' → noop (Switch is disabled)
  └─ state === 'unsupported' → noop (Switch is disabled)
```

### Initial state resolution (on mount)

```
if (!('serviceWorker' in navigator) || !('PushManager' in window)) → unsupported
else if (Notification.permission === 'denied') → denied
else
  GET /api/push-subscriptions?endpoint={currentEndpoint}
    ├─ returns active: true → enabled
    └─ returns active: false OR network error → default
```

The GET call uses `useQuery` via TanStack Query (matches existing `usePWAInstall` and query-first convention). No `useEffect` in the component — the state derivation lives in `usePushSubscription`.

### Optimistic state during subscribe/unsubscribe

Do NOT optimistically flip the Switch. Show `requesting` during the async flow and only transition on API success or failure. This prevents state desync on network errors.

## Accessibility

- Switch must have `aria-label="Ativar notificações"` when unchecked, `aria-label="Desativar notificações"` when checked.
- The helper text element should be `id`-linked via `aria-describedby` on the Switch.
- Disabled Switch states (`denied`, `unsupported`, `requesting`) must set `disabled` on the underlying Mantine `Switch` — Mantine handles `aria-disabled` automatically.
- Touch target minimum: 44×44px for the row containing the toggle.

---

## Mobile-first Constraints

Following `frontend/CLAUDE.md` "Mobile First" section:
- Validate at 375px viewport width (iPhone SE baseline) before desktop.
- The toggle row must not truncate helper text at 375px — use `wrap="wrap"` on the Group or a `Stack` for label + helper.
- Switch `size="md"` renders at a comfortable mobile tap size (Mantine default: 34px wide × 20px tall thumb).
