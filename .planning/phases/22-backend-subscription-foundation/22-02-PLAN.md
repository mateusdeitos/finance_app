<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| service → repository | userID arrives as a function argument (from auth context upstream); repository must scope writes/reads with it |
| push service → repository (Phase 23) | DeleteByEndpointAdmin prunes by endpoint alone; only invoked by trusted server-side prune logic, never from a client request |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-22-IDOR | Tampering / Elevation | `DeleteByEndpoint`, `ExistsForUser` | mitigate | Queries scoped `WHERE user_id = ? AND endpoint = ?`. userID is a caller argument derived from auth context (enforced in Plan 03 handler). Verified by acceptance criterion grep. |
| T-22-ADMIN-PRUNE | Elevation | `DeleteByEndpointAdmin` (no userID) | accept | Intentionally unscoped — server-side prune only. Not exposed via any handler in Phase 22 or 23; only callable from internal push-delivery code. Documented in interface comment. |
| T-22-DUP | Tampering | Upsert / UNIQUE(endpoint) | mitigate | `ON CONFLICT (endpoint) DO UPDATE` is race-safe and reassigns user_id, preventing duplicate device rows. |
</threat_model>
