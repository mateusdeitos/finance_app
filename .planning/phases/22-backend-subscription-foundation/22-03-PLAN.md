<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → API | Untrusted request body/query crosses here; endpoint string is client-supplied, userID is NOT |
| API → service | Handler injects userID from auth context; service trusts only that argument |
| env → process | VAPID secrets loaded at startup; must be present or the app refuses to start |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-22-IDOR | Tampering / Elevation of Privilege | Subscribe/Unsubscribe/Status handlers + service | mitigate | userID derived from `appcontext.GetUserIDFromContext`, never from request body/query; service forwards it; repository scopes `user_id = ? AND endpoint = ?`. Verified by grep acceptance criteria across handler + service. |
| T-22-KEY | Information Disclosure | main.go VAPID handling | mitigate | Startup guard names env vars but logs no key values; private key never printed/returned. Acceptance criterion greps that no key value is logged. |
| T-22-VALIDATION | Tampering / DoS | Subscribe input | mitigate | Service rejects empty endpoint / p256dh / auth with BadRequest before any DB write (ASVS V5). |
| T-22-STARTUP | Denial of Service (silent) | missing VAPID keys in prod | mitigate | `log.Fatalf` on empty keys prevents a silently-degraded deployment (RESEARCH Pitfall 3). |
| T-22-ENUM | Information Disclosure | GET status endpoint | accept | Endpoint is a 128+ char push-service URL; enumeration impractical. Rate limiting out of scope per RESEARCH Security Domain. |
</threat_model>
