## Project Overview

Go backend for a couples' finance management app. Single backend service with PostgreSQL database, deployed on Google Cloud Run.

All backend code lives in `backend/`. There is no frontend in this repo.

## Commands

All commands run from `backend/` directory. The project uses `just` as the task runner (reads `.env` automatically via `dotenv-load`).

```bash
# Run application
just run

# Tests
just test                    # all tests
just test-unit               # short tests only (go test -short)
just test-integration        # integration tests (go test -tags=integration)

# Run a single test file or package
go test ./internal/service/... -run TestTransactionService
go test ./internal/service/ -run "TestTransactionService/TestUpdate"

# Database
just up-db                   # start only PostgreSQL container
just migrate-up              # apply migrations (requires DB_DSN env var)
just migrate-create <name>   # create new .sql migration file

# Mocks
just generate-mocks          # regenerate all mocks from interfaces

# Docs
just generate-docs           # regenerate Swagger/OpenAPI docs (swag init)

# Code quality
just lint                    # run golangci-lint

# Seed
just seed <email> [partner_email]  # seed DB with test data

# Docker
just docker-up               # start all services
just docker-down             # stop all services
```

## Architecture

Layered architecture:
```
HTTP Handler (Echo) → Service (business logic) → Repository (data access) → PostgreSQL (GORM)
```

Key packages:
- `internal/domain/` — Domain models and business types used across all layers
- `internal/entity/` — ORM structs (GORM) with `ToDomain()`/`FromDomain()` conversions
- `internal/repository/` — Data access; interfaces in `repository/interfaces.go`, `Repositories` struct holds all impls
- `internal/service/` — Business logic; interfaces in `service/interfaces.go`, `Services` struct holds all impls
- `internal/handler/` — Echo HTTP handlers with Swagger annotations (`@Summary`, `@Tags`, etc.)
- `migrations/` — Goose SQL migrations (timestamped filenames)
- `mocks/` — Auto-generated by mockery (never edit manually)
- `pkg/appcontext` — Context helpers: `WithUser`/`GetUserFromContext`, `WithUserID`/`GetUserIDFromContext`
- `pkg/errors` — Custom error system: `ErrorCode` enum, `ErrorTag` for fine-grained categorization, `ServiceError` type, `ToHTTPError()` converter
- `pkg/oauth` — OAuth provider setup (Google + Microsoft via Goth); providers only enabled when ClientID/ClientSecret are set

### Dependency Injection

Services and repositories are constructed in `cmd/server/main.go` and passed through constructors. The `Services` struct depends on the `Repositories` struct. Some services (`TransactionService`, `UserConnectionService`) depend on the `Services` struct itself for cross-service calls.

### Transaction Domain

The central domain concept. Key types in `internal/domain/transaction.go`:
- `TransactionType`: `expense`, `income`, `transfer`
- `OperationType`: `credit`, `debit` (derived from TransactionType)
- `TransactionPropagationSettings`: `all`, `current`, `current_and_future` — controls how updates/deletes propagate to recurring installments
- `LinkedTransactions`: relationship table (replaced `parent_id` in Feb 2026 migration) — used to link the two sides of a transfer
- `Amount` is stored in **cents** (int64)
- `RecurrenceType`: `daily`, `weekly`, `monthly`, `yearly`; `RecurrenceSettings` holds type + repetitions OR endDate (mutually exclusive)
- `SplitSettings`: array with connection ID and percentage OR amount per split (mutually exclusive); links to user connections and destination accounts

### Other Domain Concepts

- **Settlement** (`internal/domain/settlement.go`) — Tracks debt/credit between users in shared expenses. `SettlementType`: `credit`/`debit`. Links to source and parent transactions.
- **UserConnection** (`internal/domain/user_connection.go`) — Manages relationships between two users. `UserConnectionStatusEnum`: `pending`, `accepted`, `rejected`. Has `SwapIfNeeded()` to normalize orientation, default split percentages per direction, and separate account IDs for each user.

### Middleware & Auth

- `AuthMiddleware` (`internal/middleware/`) — Extracts JWT from `Authorization: Bearer <token>` header OR `auth_token` cookie; injects `User` and `UserID` into context via `pkg/appcontext`.
- Custom error handler (`internal/middleware/error_handler.go`) — Standardized `ErrorResponse` with `error` (code) and `message` fields. Use `pkg/errors.ToHTTPError()` to convert `ServiceError` to HTTP responses.

### Testing

Integration tests use real PostgreSQL via testcontainers. The shared DB instance is initialized once via `sync.Once` in `ServiceTestWithDBSuite`.

Test suites embed `ServiceTestWithDBSuite` (defined in `internal/service/test_setup_with_db.go`), which wires up all real repositories and services against the test DB.

Helper methods on `ServiceTestWithDBSuite`: `createTestUser`, `createTestAccount`, `createTestCategory`, `createTestTag`, `createAcceptedTestUserConnection`.

When adding new repository interfaces, regenerate mocks with `just generate-mocks`.

### Key Patterns

- Repository interfaces accept `SearchOptions` or `Filter` structs rather than individual parameters
- `ComparableSearch[T]` is a generic struct for flexible query filtering (greater than, less than, equal, etc.); filter structs also support `TextSearch` and `SortBy` (field + asc/desc) with limit/offset pagination
- Soft deletes on transactions via `deleted_at` column
- `OriginalUserID` tracks which user originally created a transaction (relevant for shared/split expenses)
- Application sets `time.Local = UTC` at startup; all times are UTC
- Swagger docs are generated from handler annotations into `docs/`; run `just generate-docs` after changing handler comments
