# CLAUDE.md — Backend

This file provides guidance to Claude Code (claude.ai/code) when working with the backend.

## Project Overview

Go backend for a couples' finance management app. Single backend service with PostgreSQL database, deployed on Google Cloud Run.

## Commands

All commands run from `backend/` directory. The project uses `just` as the task runner (reads `.env` automatically via `dotenv-load`).

```bash
# Run application
just run

# Tests
just test                    # all tests
just test-unit               # short tests only (go test -short)
just test-integration        # integration tests (go test -tags=integration)

# Run a single test file or package
go test ./internal/service/... -run TestTransactionService
go test ./internal/service/ -run "TestTransactionService/TestUpdate"

# Database
just up-db                   # start only PostgreSQL container
just migrate-up              # apply migrations (requires DB_DSN env var)
just migrate-create <name>   # create new .sql migration file

# Mocks / docs
just generate-mocks          # regenerate all mocks from interfaces
just generate-docs           # regenerate Swagger docs from handler annotations

# Docker
just docker-up               # start all services
just docker-down             # stop all services
```

**Prefer `just` recipes over manual steps.** When a recipe exists for a flow, use it — don't replicate the steps by hand:

- New migration → `just migrate-create <name>` (never create the `.sql` file manually; the recipe generates the correct timestamp and scaffold).
- New/changed repository or service interface → `just generate-mocks` (never edit files under `mocks/`).
- Changed handler Swagger annotations → `just generate-docs`.
- Running the app, DB, or tests → use the matching `just` recipe before falling back to `go` / `docker` commands.
- If you think you need a manual step, check the `justfile` first.

## Architecture

Layered architecture:

```
HTTP Handler (Echo) → Service (business logic) → Repository (data access) → PostgreSQL (GORM)
```

Key packages:

- `internal/domain/` — Domain models and business types used across all layers
- `internal/entity/` — ORM structs (GORM) with `ToDomain()`/`FromDomain()` conversions
- `internal/repository/` — Data access; interfaces in `repository/interfaces.go`, `Repositories` struct holds all impls
- `internal/service/` — Business logic; interfaces in `service/interfaces.go`, `Services` struct holds all impls
- `internal/handler/` — Echo HTTP handlers
- `internal/middleware/` — Echo middleware (auth, logger, error handler)
- `internal/config/` — Typed configuration loaded from env
- `migrations/` — Goose SQL migrations (timestamped filenames)
- `mocks/` — Auto-generated by mockery (never edit manually)
- `pkg/` — Public utilities (`appcontext`, `applog`, `errors`, `database`, `oauth`, `tests`)

### Dependency Injection

Services and repositories are constructed in `cmd/server/main.go` and passed through constructors. The `Services` struct depends on the `Repositories` struct. Some services (`TransactionService`, `UserConnectionService`, `ChargeService`) depend on the `Services` struct itself for cross-service calls.

When adding a new service/repository:

1. Define the interface in `service/interfaces.go` or `repository/interfaces.go`.
2. Implement with a lowercase struct name (e.g. `transactionService`) and pointer receivers.
3. Wire it in `cmd/server/main.go` into the `Repositories`/`Services` struct.
4. Run `just generate-mocks`.

## Error Handling (`pkg/errors`)

Central type: `ServiceError{ Code, Message, Err, Tags }`. Always return `*ServiceError` from services — handlers convert to HTTP via `pkgErrors.ToHTTPError(err)`.

Error codes (map 1:1 to HTTP status):
`NOT_FOUND` (404), `ALREADY_EXISTS` (409), `UNAUTHORIZED` (401), `FORBIDDEN` (403), `VALIDATION_ERROR` / `BAD_REQUEST` (400), `INTERNAL_ERROR` (500).

Constructors:

- Helpers: `NotFound(resource)`, `AlreadyExists(resource)`, `Forbidden(msg)`, `Unauthorized(msg)`, `Validation(msg)`, `BadRequest(msg)`, `Internal(msg, err)`.
- Low-level: `New(code, msg)`, `Wrap(code, msg, err)`, `NewWithTag(code, tags, msg)`.
- Detection: `IsServiceError`, `AsServiceError`, `IsNotFound`.

`ErrorTag` is a typed enum (`type ErrorTag string`) with the convention `DOMAIN.WHAT_HAPPENED` — e.g. `TRANSACTION.AMOUNT_MUST_BE_GREATER_THAN_ZERO`, `CATEGORY.DUPLICATE_NAME`. To add a new validation error:

1. Add an `ErrorTagXxx` constant in `pkg/errors/errors.go`.
2. Declare a sentinel `ErrXxx = NewWithTag(ErrCodeBadRequest, []string{string(ErrorTagXxx)}, "...")` next to it.
3. For errors inside a slice (split settings, tags, etc.), use a closure:
   ```go
   ErrSplitSettingInvalidConnectionID = func(index int) *ServiceError {
       return NewWithTag(ErrCodeBadRequest, []string{string(ErrorTagSplitSettingInvalidConnectionID)}, fmt.Sprintf("... at index %d", index)).AddIndex(index)
   }
   ```

`ToHTTPError` produces a `*TaggedHTTPError{Code, Message, Tags}`; the `ErrorHandler` middleware serializes it as:

```json
{
  "error": "Bad Request",
  "message": "...",
  "tags": ["TRANSACTION.AMOUNT_MUST_BE_GREATER_THAN_ZERO"]
}
```

The frontend consumes `tags` to render localized messages — keep tag names stable across releases.

## Middleware (`internal/middleware`)

- **`LoggingMiddleware(globalLogger)`** — generates a UUID v4 `request_id`, sets the `X-Request-ID` response header, injects a per-request `zerolog.Logger` into context via `applog.WithLogger`, and emits **exactly one** JSON log line at the end of the chain. Log level is derived from the response status (2xx→info, 4xx→warn, 5xx→error).
- **`NewAuthMiddleware(...).RequireAuth`** — validates a JWT from the `Authorization` header or cookie and injects the user into context with `appcontext.WithUserID`. Apply at the `/api` group in `cmd/server/main.go`.
- **`ErrorHandler(err, c)`** — Echo's `HTTPErrorHandler`. Converts `*TaggedHTTPError` → JSON response (`middleware.ErrorResponse`), falls back to `*echo.HTTPError` or a generic 500. Also enriches the request logger with `error_code`, `error_message`, and `error_tags` before the logging middleware emits the final line.

## Logging (`pkg/applog`)

One logger per request, mutated as it flows through the chain. Use:

```go
applog.FromContext(c.Request().Context()).
    With("user_id", userID).
    With("transaction_id", id)
```

Rules:

- **Do not** create a new `zerolog.Logger` inside a handler or service — always fetch from context.
- **Do not** call `.Msg(...)` yourself; the logging middleware emits the single final line with status and latency.
- Use `.With(key, value)` to accumulate structured fields that will show up on the request's log line.

## Database transactions (`internal/repository/db_transaction.go`)

A `DBTransaction` interface (`Begin/Commit/Rollback`) stores the active `*gorm.DB` inside `context.Context`. Every repository method uses `GetTxFromContext(ctx, r.db)` so it automatically joins the caller's transaction when one is active.

Service pattern for multi-step writes:

```go
ctx, err := s.tx.Begin(ctx)
if err != nil { return nil, err }
defer s.tx.Rollback(ctx) // no-op if already committed

if err := s.repo.DoA(ctx, ...); err != nil { return nil, err }
if err := s.repo.DoB(ctx, ...); err != nil { return nil, err }

if err := s.tx.Commit(ctx); err != nil { return nil, err }
return result, nil
```

Pre-mock `Begin/Commit/Rollback` with `.Maybe()` in unit tests (see `test_setup.go`).

## Routing & handlers

Routes are registered in `cmd/server/main.go` using Echo groups. Authenticated routes live under `api := e.Group("/api")` with `api.Use(authMiddleware.RequireAuth)`; subgroups organize routes by resource.

Handler layer responsibilities (keep thin):

1. Extract user id: `userID := appcontext.GetUserIDFromContext(c.Request().Context())`.
2. Bind request: `c.Bind(&dto)`; return `echo.NewHTTPError(http.StatusBadRequest, "...")` on parse error.
3. Parse path/query params (e.g. `strconv.Atoi(c.Param("id"))`).
4. Call the service with `c.Request().Context()`.
5. On error: `return pkgErrors.ToHTTPError(err)`.
6. On success: `return c.JSON(status, payload)`.

**Business validation belongs in the service**, not the handler. Services return `*ServiceError` with tags; handlers must not re-translate error messages.

## Configuration (`internal/config/config.go`)

Env-based, loaded via `godotenv` (missing `.env` is not fatal). Typed sub-structs: `ServerConfig`, `DatabaseConfig`, `JWTConfig`, `OAuthConfig`, `AppConfig`. Provide sensible defaults for dev. When adding new config, extend `Config` with a new sub-struct and wire it into DI in `main.go`.

## Swagger / OpenAPI

Handler methods carry godoc annotations consumed by `swag`. Minimum set:

```go
// Create godoc
// @Summary      Create transaction
// @Description  ...
// @Tags         transactions
// @Accept       json
// @Security     CookieAuth
// @Security     BearerAuth
// @Param        transaction  body  domain.TransactionCreateRequest  true  "Transaction data"
// @Success      201
// @Failure      400  {object}  middleware.ErrorResponse
// @Failure      401  {object}  middleware.ErrorResponse
// @Router       /api/transactions [post]
```

All error `@Failure` entries reference `middleware.ErrorResponse`. Run `just generate-docs` after changing annotations — the frontend consumes the regenerated spec in `backend/docs/`.

## Migrations (`migrations/`)

Goose-style SQL files. Naming: `YYYYMMDDHHMMSS_description.sql`. Each file contains two blocks:

```sql
-- +goose Up
CREATE TABLE ...;
CREATE INDEX ...;

-- +goose Down
DROP INDEX ...;
DROP TABLE ...;
```

Always write a symmetric `Down` block. **Create migrations with `just migrate-create <name>`** — do not hand-write the file or its timestamp.

## Transaction domain

The central domain concept. Key types in `internal/domain/transaction.go`:

- `TransactionType`: `expense`, `income`, `transfer`
- `OperationType`: `credit`, `debit` (derived from `TransactionType`)
- `TransactionPropagationSettings`: `all`, `current`, `current_and_future` — controls how updates/deletes propagate to recurring installments
- `LinkedTransactions`: relationship table (replaced `parent_id` in Feb 2026 migration) — used to link the two sides of a transfer
- `Amount` is stored in **cents** (`int64`)

## Testing

Two suites, both embedding `testify/suite.Suite`:

- **`ServiceTestSuite`** (`internal/service/test_setup.go`) — unit tests with mockery-generated mocks; fast; runs with `just test-unit`.
- **`ServiceTestWithDBSuite`** (`internal/service/test_setup_with_db.go`) — integration tests against real PostgreSQL via testcontainers. Naming convention: types include `WithDB`.

Integration DB setup:

- Container started once per process via `sync.Once`; goose migrations applied at startup.
- No truncate/rollback between tests — isolation via **randomized data** (e.g. `math/rand.Int64()` in emails/names). Design new helpers the same way.
- Short-mode guards (`if testing.Short() { t.Skip(...) }`) keep DB-heavy tests out of `just test-unit`.

Helper methods on `ServiceTestWithDBSuite`: `createTestUser`, `createTestAccount`, `createTestCategory`, `createTestTag`, `createAcceptedTestUserConnection`, `createManyConnections`. Reuse and extend these instead of creating ad-hoc fixtures.

Mocks:

- Generated by mockery; located under `mocks/` with `DO NOT EDIT` headers.
- Use the fluent API: `mock.EXPECT().Method(args).Return(...).Once()` / `.Maybe()` / `.RunAndReturn(...)`.
- `DBTransaction.Begin/Commit/Rollback` are pre-mocked with `.Maybe()` in `test_setup.go`.
- Regenerate with `just generate-mocks` whenever an interface changes.

Handler tests (`internal/handler/*_test.go`):

- Inject the authenticated user with `appcontext.WithUserID(ctx, userID)`.
- Use `httptest.NewRecorder()` and assert on status code + `errors.As` for typed service errors.

## Code conventions

- **Files**: `snake_case.go`.
- **Types**: `PascalCase`. Interfaces named after the role (e.g. `TransactionService`, `UserRepository`) — **no** `-er` or `Interface` suffix.
- **Implementations**: lowercase struct (`transactionService`), pointer receivers everywhere.
- **`context.Context`** is always the **first** parameter in service and repository methods.
- **Enums**: `type Foo string` + `const FooBar Foo = "bar"` + method `(f Foo) IsValid() bool`. Add helpers like `Invert()` when they exist in the domain.
- **Times**: always UTC (`time.Date(..., time.UTC)`). `time.Local` is set to UTC at startup — do not rely on local time anywhere.
- **Money**: always `int64` cents end-to-end (domain, entity, DTO, API).
- **Search/filter**: accept a `*XxxSearchOptions` or `*XxxFilter` struct instead of variadic params. Reuse the generic `ComparableSearch[T]` for numeric/date ranges.
- **Domain ↔ Entity conversion**: `entity.XxxFromDomain(d)` and `(e *Xxx) ToDomain()`. Never leak GORM types past the repository.
- **Error wrapping**: return `*ServiceError` from services; use `Wrap(code, msg, err)` or `Internal(msg, err)` to preserve the underlying cause.

## `pkg/` packages (public utilities)

- `pkg/appcontext` — typed context keys (e.g. `WithUserID`, `GetUserIDFromContext`).
- `pkg/applog` — per-request zerolog logger (`WithLogger`, `FromContext`).
- `pkg/errors` — `ServiceError`, `ErrorTag`, `TaggedHTTPError`, HTTP mapping.
- `pkg/database` — GORM bootstrap.
- `pkg/oauth` — third-party OAuth clients (Google, etc.).
- `pkg/tests` — testcontainers helpers (`test_database.go`).

## Key patterns (summary)

- Repository interfaces accept `SearchOptions`/`Filter` structs rather than individual parameters.
- `ComparableSearch[T]` is a generic struct for flexible query filtering (gt, lt, eq, etc.).
- Soft deletes on transactions via `deleted_at` column.
- `OriginalUserID` tracks which user originally created a transaction (relevant for shared/split expenses).
- Multi-repo writes go through `DBTransaction` so the same tx flows via context.
- Always convert service errors to HTTP with `pkgErrors.ToHTTPError(err)` at the handler boundary.

## Business rules

### Shared transactions

#### Shared expenses

When users create a shared expense, 3 transactions are created:

- 1 transaction for the author in its private account with an operation type = 'debit' with the whole amount of the transaction
- 1 settlement for the author in its shared account from the user_connection with the amount being the amount set on the split setting
- 1 transaction for the other end user in its shared account from the user_connection with an operation type = 'debit' with the amount being the amount set on the split setting

Transactions created in the shared accounts and related settlements should ALWAYS match amount, meaning that if the main transaction amount is updated or the split_setting, both should be updated.

Example:

We have 2 users: User A (id=1) and User B (id=2), both are connected

The connection has:

- from_user_id: 1
- from_account_id: 10
- to_user_Id: 2
- to_account_id: 20

User A create a shared expense with the amount of R$ 100,00, splitting it in 50% with User B, then:

- An expense is created on whatever account User A chooses with the amount of R$ 100,00 and operation_type = 'debit'
- A settlement is created on account_id = 10 with the amount of R$ 50,00 with type = 'credit'
- An expense is created on account_id = 20 with the amount of R$ 50,00 with the operation_type = 'debit'

When User A filter transactions by account_id = 10 it will see only a settlement of R$ 50,00 being credited and the balance for that account in that MM/YYYY will be +R$50,00
When User B filter transactions by account_id = 20 it will see only an expense of R$ 50,00 being debited and the balance for that account in that MM/YYYY will be -R$50,00

#### Shared incomes

It's the same as expenses but with operation type flipped

#### Transfers between different users

When users create a transfer to another user through a connection account, 3 transactions are created:

- 1 transfer operation type = 'debit' with the account being the author private account
- 1 transfer operation type = 'credit' with the account being the author side's account of the user_connection
- 1 transfer operation type = 'credit' with the account being the other user side's account of the user_connection

### Balance

The previous rules are created in a way that when we fetch the balance of a month we:

- sum incomes
- subtract expenses
- subtract debit transfers
- sum credit transfers
- sum credit settlements (conditionally to hide_settlements params)
- subtract debit settlements (conditionally to hide_settlements params)

### Charges

Charges are entitites created to group transfers used to level the balance of shared accounts.

In the example of the shared expense, User A had a balance of +50,00 and User B a balance of -50,00.

If User A create a charge as a charger he can charge the 50,00 that User B is due.

User B can also initiate the process as a payer and create a charge to pay User A 50,00.

Users can also set arbitrary amounts to charges, thats not an issue.

When a charge is accepted we create a transfer on both users to credit/debit the shared accounts and make the balance 0 (if the amount is exactly what the user is due to one another)
